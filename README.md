# 1c_AI

ИИ-ассистент поверх базы 1С Розница: чат-бот внутри 1С обращается к локальной LLM
(через n8n), которая формирует и выполняет запросы на языке запросов 1С,
опираясь на схему конкретной конфигурации, проиндексированную в векторной базе.

## Архитектура

```
1С (ЧатБот.epf) → n8n (webhook /chat-1c) → AI Agent (Ollama, qwen2.5:7b)
                                                ├── query_1c            — HTTP-запрос к 1С
                                                └── Schema Search Tool  — поиск по схеме конфигурации в Qdrant
```

Два независимых процесса:

1. **Indexing pipeline** — выгружает метаданные конфигурации 1С в JSON, режет на
   объекты, считает эмбеддинги (Ollama, `bge-m3`) и записывает в Qdrant.
   Запускается вручную при изменении конфигурации.
2. **Chat workflow** — основной цикл: 1С → n8n → AI Agent → (поиск по схеме →
   формирование запроса → выполнение запроса) → ответ.

Retrieval реализован как отдельный вызываемый sub-workflow (`Schema Search Tool`),
подключённый к AI Agent как tool — это классическая схема RAG (индексация +
retrieval), просто база знаний — не документы, а схема конфигурации 1С.

## Стек

- **n8n** — оркестрация workflow, self-hosted, Basic Auth
- **Ollama** — локальные модели, работает нативно на хосте (не в контейнере)
  - `qwen2.5:7b` — чат-модель для агента
  - `bge-m3` — модель эмбеддингов (1024-мерный вектор, контекст до 8192 токенов)
- **Qdrant** — векторная база для схемы конфигурации
- **1С:Предприятие** (Розница) — внешняя обработка `ЧатБот.epf` + HTTP-сервис,
  принимающий запрос на языке 1С и возвращающий результат

## Структура репозитория

```
docker-compose.yml         # n8n + Qdrant
Makefile                   # make export-workflows — выгрузка workflow из n8n в git
ВыгрузкаМетаданныхJSON.bsl # обработка 1С: выгружает метаданные конфигурации в JSON
ЧатБот.epf                 # внешняя обработка 1С — чат-клиент
workflows/                 # экспортированные n8n workflow (читаемый JSON)
system_prompt.txt          # системный промпт AI Agent
n8n_shared/                 # (не в git) — общая папка n8n ↔ хост, .n8n-files
n8n_data/                   # (не в git) — служебные данные n8n (SQLite, credentials)
qdrant_data/                 # (не в git) — данные Qdrant
```

`n8n_data` и `qdrant_data` — runtime-данные, не код; исключены из git через
`.gitignore`. Если у вас в истории репозитория они закоммичены — прогоните:
```bash
git rm -r --cached n8n_data qdrant_data
```

## Запуск

```bash
cp .env.example .env   # заполнить N8N_BASIC_AUTH_USER / N8N_BASIC_AUTH_PASSWORD
docker compose up -d
```

n8n будет доступен на `http://localhost:5678`, Qdrant — на `http://localhost:6333`.

Ollama должна быть установлена и запущена **на хосте** (не в контейнере) —
контейнеры обращаются к ней через `host.docker.internal`. Модели:
```bash
ollama pull qwen2.5:7b
ollama pull bge-m3
```

## Индексация схемы конфигурации

1. В 1С выполнить обработку `ВыгрузкаМетаданныхJSON.bsl` → получить JSON-файл
   со всеми объектами метаданных
2. Положить файл в `n8n_shared/` (примонтирована в контейнер n8n как
   `/home/node/.n8n-files`)
3. Запустить workflow-индексатор в n8n вручную (кнопка Execute Workflow)

Индексатор делает полную переиндексацию: удаляет и заново создаёт коллекцию
`schema_1c` в Qdrant, затем для каждого объекта метаданных считает эмбеддинг и
пишет точку с payload. ID точек — случайные UUID, детерминированность не
нужна, так как переиндексация всегда полная, а не инкрементальная.

Технические объекты (`*ПрисоединенныеФайлы`, `ВерсииОбъектов*`,
`ХранилищеНастроек*`) отфильтровываются на этапе построения текста для
эмбеддинга — список паттернов исключения живёт в коде узла фильтрации и
расширяется по мере обнаружения нового шума в выдаче поиска.

## Экспорт workflow в git

```bash
make export-workflows
```
Экспортирует все workflow n8n в `workflows/*.json`, файлы называются по
имени workflow (не по внутреннему ID).

## Известные технические ограничения

- **Ollama + встроенная графика**: по умолчанию Ollama использует только CPU
  (`OLLAMA_IGPU_ENABLE=1` в конфиге systemd включает Intel iGPU и заметно
  ускоряет инференс)
- **Контекстное окно чат-модели**: `qwen2.5:7b` в Ollama по умолчанию
  ограничена 4096 токенами — задаётся явно (`Context Length` в ноде Ollama
  Chat Model), иначе длинный системный промпт + результаты поиска по схеме
  могут не поместиться, и модель вернёт пустой ответ без явной ошибки
- **Имена tool-нод для Ollama**: `toolWorkflow`-нода генерирует имя функции
  для LLM из имени самой ноды на холсте — имя должно быть без пробелов и
  кавычек (`schema_search_tool`, а не `Call 'Schema Search Tool'`), иначе
  Ollama может молча возвращать пустую генерацию на весь запрос с `tools`
- **Эмбеддинги на CPU**: массовая индексация (тысячи объектов) может
  провоцировать рост потребления памяти Ollama при долгой последовательности
  запросов — в пайплайне это компенсируется периодической выгрузкой модели
  (`keep_alive: 0`) каждые N объектов
- **Контекст `nomic-embed-text`**: реальный лимит около 2048 токенов
  независимо от заявленных 8192 — при необходимости длинного контекста для
  эмбеддингов используется `bge-m3` (честные 8192)
